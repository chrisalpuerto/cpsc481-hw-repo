## Analysis: Default Tree vs. Constrained Tree

The default tree grows deep branches that split on noisy, low-signal features — `Price_$$$`, `Type_Italian`, `Raining_Yes`, and `Reservation_Yes` — suggesting the model memorized quirks in the training data rather than learning a general rule. The constrained tree (`max_depth=3`, `min_samples_split=10`) forces the algorithm to commit only to the most statistically supported splits, leaving a much shallower structure built around three features: `Patrons_Some`, `WaitEstimate`, and `Bar_Yes`. This maps well onto human intuition: check how busy the restaurant is, estimate how long the wait is, and note whether there's a bar to wait at if needed. The default tree's inclusion of restaurant type (`Type_Italian`) and whether it's raining as decision factors feels arbitrary and hard to justify in a real-world context, while the constrained tree's logic is easy to explain and defend. For a person deciding whether to wait for a table, the constrained tree reads almost like common sense, whereas the default tree reads like an overcomplicated flowchart that would be impractical to follow.
